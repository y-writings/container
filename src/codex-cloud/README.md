# Codex Cloud セットアップ方針

このディレクトリは、**Codex Cloud（Ubuntu ベース）上で再現可能な開発環境**を構築するための設定を管理します。

- パッケージ管理: Nix flake
- Nix インストール: Determinate Systems のインストーラー
- 実行エントリ: `src/codex-cloud/setup.sh`

## 1) Codex Cloud 上で手動実施すべき事前設定

以下は **GitHub リポジトリで管理しない** ことを推奨します。

- Codex Cloud の Web UI / コンソールで行う初期設定
  - ワークスペース作成時のオプション選択
  - ランタイム権限や組織ポリシーに依存する設定
- シークレット・認証情報の設定
  - GitHub Token、クラウド API キー、SSH 鍵、各種資格情報
  - `.env` 相当の秘匿値や個人固有のアクセストークン
- 人や組織、環境に依存する値の投入
  - 個別アカウント ID、課金/監査に関わる設定

### なぜリポジトリ管理しないのか

- **秘匿性**: シークレットを Git に含めると漏えいリスクが高い。
- **環境依存性**: UI 操作や組織ポリシー依存の設定はコード化しづらい。
- **責務分離**: 監査対象（秘密情報）と再現可能な構成（コード）を分離できる。

## 2) GitHub リポジトリで管理すべき設定・スクリプト

以下は **再現性とバージョン管理の観点から GitHub で管理** すべきです。

- `apt` によるベースパッケージ導入
  - `curl`, `ca-certificates`, `xz-utils` など、Nix 導入や実行に必要な共通基盤
- Determinate Systems の Nix インストール手順
  - `install` スクリプトの呼び出し方法とオプション
- `flake.nix` / `flake.lock` による環境定義
  - 依存パッケージ群を宣言的に管理
  - `nix profile install .#devtools` でツール群を再現可能に導入
- 自動化スクリプト（`setup.sh`）
  - 実行順序・前提チェック・冪等スキップ・ログ出力を明文化

## 3) 分水嶺となる判断基準（手動 vs リポジトリ管理）

以下の判断軸で決めます。

1. **冪等性**
   - 何度実行しても同じ結果になる処理はスクリプト化して Git 管理。
   - 一度限り/人手確認が必要な操作は手動側。
2. **再現性**
   - 他の開発者・新規環境で同じ状態を再現したいものは Git 管理。
3. **秘匿性**
   - シークレットや個人情報は Git 管理しない（Secret Manager / 環境変数で注入）。
4. **環境依存性**
   - 組織ポリシー・契約・UI 状態に依存しやすいものは手動実施。
5. **監査容易性**
   - 変更履歴を追跡したい構成・手順はコード化して PR でレビューする。

## 4) セットアップ実行手順

```bash
bash src/codex-cloud/setup.sh
```

`setup.sh` は以下を順番に実行します。

1. `apt` ベースパッケージ導入（不足分のみ）
2. Determinate Systems インストーラーで Nix 導入（導入済みならスキップ）
3. flake 有効化チェックと `nix profile install .#devtools`
4. `src/codex-cloud/` 配下の実行可能ファイルをファイル名昇順で実行
   - `setup.sh` 自身は除外
   - 想定例: `10-*.sh` → `20-*.sh` の順で段階実行
